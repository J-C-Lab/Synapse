# Synapse Plugin Development

This file gives coding agents and contributors the local rules for this plugin
project. Keep changes small, buildable, and aligned with the Synapse plugin
runtime.

## Project Shape

- `synapse.json` is the plugin manifest. It declares the plugin id, version,
  engine compatibility, capabilities, commands, tools, and entry file.
- `src/index.ts` exports the plugin module. It should use the types from
  `@synapsepkg/plugin-sdk`.
- `dist/` is generated by the build command and should not be edited by hand.
- The installable package is a `.syn` file produced by `npm run build`.

## Runtime Rules

- Export a CommonJS plugin module with `export = plugin`.
- Do not import React or render DOM directly. Commands return declarative
  views and the Synapse host renders them.
- Avoid Node built-ins and filesystem/network side effects unless the runtime
  explicitly supports them.
- Keep requested capabilities in `synapse.json` as narrow as possible.
- If a command id changes, update both `synapse.json` and `src/index.ts`.
- If a tool shape changes, update its manifest schema and implementation
  together.

## Capabilities

`manifest.capabilities` is a list of `{ id, scope? }` entries — this is the
**only** access a plugin gets to the OS/network/user data; there is no
separate "permissions" field. Every capability has a fixed tier the host
enforces, not something you choose:

- `auto` — granted automatically (e.g. `storage:plugin`, `notification`).
- `consent` — the user approves once per install (e.g. `clipboard:read`,
  `fs:read`, `system:open-url`).
- `elevated` — the user approves per call, or per narrower scope (e.g.
  `network:https`, `fs:write`, `hotkey:global`, `clipboard:watch`).

Scoped capabilities (`network:https`, `fs:read`/`fs:write`/`fs:watch`,
`hotkey:global`, `credentials:broker`) also take a `scope` narrowing exactly
what the capability covers (an allowed host, a path prefix, an accelerator).
Declare the narrowest scope the plugin actually needs — the host rejects a
call outside the declared scope even if the capability itself is granted.
Only ever call host APIs through `ctx` (`ctx.network.fetch`, `ctx.fs.*`,
`ctx.clipboard.*`, `ctx.credentials.*`) — never reach for Node's own `fetch`,
`fs`, or `child_process` directly; those aren't capability-checked and won't
work the same way (or at all) inside the plugin sandbox.

If the plugin needs to act on a third-party service on the user's behalf
(OAuth), declare a `credentials:broker` capability plus an entry under
`contributes.credentials` in the manifest — the host handles the OAuth flow
and injects the token at the network layer; the plugin code never sees the
raw token.

If the plugin should run in the background without the user opening it (a
schedule, a hotkey, a filesystem change), declare a trigger under
`contributes.triggers` and export the matching handler under `triggers` in
`src/index.ts`. Unlike commands and tools, **nothing validates that the
`handler` name actually matches an export** — neither `synapse-plugin build`/
`validate` nor plugin load time. A mismatch registers the trigger successfully
and then silently does nothing when it fires, with no error anywhere.
Double-check the manifest's `handler` string and the exported function name
match exactly, and confirm the trigger actually fires once during development.

## Commands And Tools

- Commands belong under `contributes.commands` in `synapse.json` and under the
  `commands` map in `src/index.ts`.
- Tools belong under `contributes.tools` in `synapse.json` and under the
  `tools` map in `src/index.ts`.
- Tool inputs are validated against `inputSchema` before the implementation is
  called. Keep schemas explicit and return structured data when useful.
- Use `annotations.readOnlyHint: true` for tools that do not mutate user data.

## Development

```bash
npm install
npm run dev
npm run build
npm run validate
```

Use `npm run dev` while working with a running Synapse app. Use `npm run build`
before sharing or releasing a plugin.

## Release

The generated GitHub Actions workflow builds the plugin on version tags and
uploads the `.syn` artifact plus its SHA-256 file. Keep release outputs in sync
with the package extension used by Synapse.
