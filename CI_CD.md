# CI/CD

This repo is built around GitHub Actions and local pnpm checks.

## Local checks

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm eval
pnpm build
pnpm electron:build:win
pnpm test:e2e:packaged
```

See [TESTING.md](TESTING.md) for the full Checkpoint R gate checklist.

## What CI should enforce

- formatting and linting
- TypeScript type safety
- workspace SDK build output generated during CI (the SDK `dist/` is not committed)
- unit tests and eval suite
- Windows x64 Electron build smoke checks (packaged renderer + MCP readiness)

## Publication profile (Checkpoint R)

**Public releases currently ship Windows x64 only** — NSIS (`.exe`) and MSI (`.msi`) installers,
plus the electron-updater feed. The release pipeline does **not** build, verify, or publish macOS or
Linux artifacts.

| Scope                         | Supported                                              | Notes                                                      |
| ----------------------------- | ------------------------------------------------------ | ---------------------------------------------------------- |
| CI release build              | Windows x64                                            | `build-electron.yml` on `windows-latest`                   |
| GitHub Release assets         | Windows x64 NSIS + MSI + feed                          | Assembled by `release-admission-gate.mjs`                  |
| Auto-update feed              | `latest.yml` (Windows only)                            | No `latest-linux.yml` or macOS feed                        |
| Local packaging for release   | `pnpm electron:build:win`                              | Required before `pnpm test:e2e:packaged`                   |
| Local packaging (unsupported) | `pnpm electron:build:mac`, `pnpm electron:build:linux` | Convenience scripts only — not part of the release profile |

Runtime stack for this profile: **Electron 43.x**, **electron-builder 26.x**, **electron-updater 6.8.9**.
Repository/CI tooling itself runs on Node 22.13.x; only the packaged Electron main process (and the
`ELECTRON_RUN_AS_NODE=1` MCP child it spawns) run under Electron 43's embedded Node 24.x.

## Release flow

Releases are triggered by Git tags, not by release branches. Tag pushes are the **sole publication
route** — manual `workflow_dispatch` is dry-run only and never creates a GitHub Release.

The release workflow runs when a tag matching `v*` is pushed:

```text
v0.3.0
v1.0.0
v1.2.3-beta.1
```

Recommended release steps:

1. Make sure `main` is green and contains the code to release.
2. Create a release pull request that updates `package.json` version.
3. Merge the release pull request into `main`.
4. Pull the latest `main` locally.
5. Run a dry-run proof on the release commit (optional but recommended before tagging).
6. Create and push an annotated tag.

Example:

```bash
git switch main
git pull
git tag -a v0.3.0 -m "v0.3.0"
git push origin v0.3.0
```

After the tag is pushed, GitHub Actions will:

1. run quality checks
2. run tests
3. build the Windows x64 Electron artifacts
4. run the release admission gate (verify, assemble, attest)
5. create a **draft** GitHub Release from the approved bundle only

Review the draft release on GitHub, check the generated notes and artifacts, then publish it manually.

Do not create `v0.3.0` as a branch. Version-like names are reserved for tags.

### Windows release assets

The admission gate assembles exactly these files into `release-approved-bundle/assets/` and uploads
them to the GitHub Release:

**Installers and updater feed**

- `Synapse Setup X.Y.Z.exe` — NSIS installer
- `Synapse Setup X.Y.Z.exe.blockmap` — differential-update blockmap (gzip JSON)
- `latest.yml` — electron-updater feed (references the NSIS `.exe` by name, sha512, and size)
- `Synapse X.Y.Z.msi` — MSI installer

**Proof files** (generated by the admission gate, not produced by electron-builder directly)

- `release-profile.json` — machine-readable publication profile (`windows` / `x64` / `nsis` + `msi`)
- `release-context.json` — commit SHA, workflow run ID, and mode (`release` or `dry-run`)
- `signing-status.json` — per-platform signing claim (currently `unsigned-unverified` for Windows)
- `manifest.json` — sha512 inventory of every asset file

The release body is rendered from these proof files and includes a GitHub artifact attestation link.

### Dry-run proof (manual dispatch — never publishes)

To prove the Windows release pipeline without creating a GitHub Release, dispatch the Release
workflow in dry-run mode from a branch or commit:

```bash
gh workflow run release.yml --ref <branch-or-sha> -f dry_run=true
```

Constraints:

- `workflow_dispatch` **must** set `dry_run=true`; any other value is rejected.
- Dry-run skips tag-version and eval-signal checks (no tag exists yet).
- Dry-run **never** runs the `create-release` job — no draft or published release is created.

**Permanent attestation residue** from a successful dry-run:

- Workflow run summary titled "Dry-run release proof" (commit SHA, package version, attestation URL)
- GitHub artifact attestation on `release-approved-bundle/assets/*`
- Artifact `release-approved-bundle-dry-run` (90-day retention) containing the same proof bundle
  as a real release, with `release-context.json` `"mode": "dry-run"` and a release body prefixed
  `DRY RUN — NOT A RELEASE`

Do **not** dispatch dry-run on `main` as part of routine CI — use a feature branch or release PR
commit instead.

### Tag release gates (eval + draft-first)

Tag pushes run in `RELEASE_MODE=release`. Before the admission gate approves the bundle:

1. **Tag/version match** — the pushed tag (`vX.Y.Z`) must match `package.json`'s `version`.
2. **Eval signal** — the most recent completed `eval-nightly.yml` run must be `success`, its
   `eval-nightly-status-json` artifact must report `state: "clean"`, and its `headSha` must match
   the commit being released (and be no older than 48 hours). If the signal is missing or stale,
   run `gh workflow run eval-nightly.yml --ref <tag>` and retry once it completes clean.

After the gate passes, `create-release` uploads assets and opens a **draft** GitHub Release. Users
do not see updates until you manually publish the draft (see auto-update below).

### Cutting a release — step-by-step checklist

Replace `X.Y.Z` with the new version throughout. The tag must be `vX.Y.Z` and must match
`package.json`'s `version`.

**1. Pre-flight (on `main`)**

- [ ] `main` is green: run the Checkpoint R local gate in [TESTING.md](TESTING.md).
- [ ] If the AI layer changed, the AI smoke checklist in [TESTING.md](TESTING.md) passed.
- [ ] `package.json` `version` is bumped to `X.Y.Z` (via a release PR) and merged to `main`.
- [ ] Locally: `git switch main && git pull`, working tree clean.
- [ ] (Recommended) Dry-run proof green on the release commit: `gh workflow run release.yml --ref main -f dry_run=true`.

**2. Tag and build**

- [ ] Tag and push (see the example below).
- [ ] `eval-nightly.yml` is clean for the tagged commit (gate enforces this automatically).
- [ ] The **Release** workflow run for the tag is green (quality → test → build-electron →
      release-admission-gate → create-release).

```bash
git tag -a vX.Y.Z -m "vX.Y.Z"
git push origin vX.Y.Z
```

**3. Review and publish the draft**

- [ ] Open the **draft** GitHub Release. Confirm the generated notes look right.
- [ ] Confirm the Windows assets only:
  - `Synapse Setup X.Y.Z.exe`, `Synapse Setup X.Y.Z.exe.blockmap`, `latest.yml`, `Synapse X.Y.Z.msi`
  - Proof files: `release-profile.json`, `release-context.json`, `signing-status.json`, `manifest.json`
  - Attestation link in the release body resolves
- [ ] **Publish** the release (remove draft). Auto-update only reads **published** releases, so it
      stays invisible to users until this step.

**4. Post-publish verification**

- [ ] Download the NSIS or MSI installer onto a clean Windows machine and confirm the app launches
      (dismiss the SmartScreen "unknown publisher" prompt — builds are unsigned).
- [ ] **Auto-update upgrade smoke (Windows only)**: with the _previous_ version installed and
      running, confirm it shows the update banner → **Download** → **Restart to update** → relaunches
      on `X.Y.Z`. (Easiest to verify on the release _after_ the first one that shipped `latest.yml`.)

**Rollback**

- [ ] If a release is bad, **unpublish or delete it** (including its `latest.yml`) so the updater
      stops offering it, then ship a fixed, higher version. electron-updater never moves users
      backwards, so you cannot "downgrade" via a lower tag — always roll forward.

## Auto-update (electron-updater)

The app checks for updates on startup and surfaces an in-app banner: the user clicks **Download**,
then **Restart to update**. The Windows flow is:

```text
GitHub Release → latest.yml → download unsigned NSIS installer
→ verify SHA-512 → quit Synapse → run the installer in place → reopen Synapse
```

`electron-updater` performs the SHA-512 check against the value in `latest.yml`; a missing or
mismatched checksum fails before installation. `quitAndInstall(false, true)` explicitly requests
that NSIS reopen the upgraded app after installation. The flow lives in
[src/main/updates/update-service.ts](src/main/updates/update-service.ts) (state machine, unit-tested),
wired to electron-updater in [src/main/index.ts](src/main/index.ts) and shown by
[update-banner.tsx](src/renderer/src/components/update-banner.tsx).

**Auto-update runs on packaged Windows only.** macOS and Linux are not built or published by the
current release pipeline, so there is no updater feed for those platforms. Do not claim macOS or
Linux auto-update in release notes or user-facing docs.

The update feed is the project's **GitHub Releases** (configured under `build.publish` in
`package.json`). For the updater to find a new version, the published GitHub Release must contain:

- `latest.yml`, `*.exe`, `*.exe.blockmap`

The release workflow uploads these via the admission gate (see `build-electron.yml` and
`release.yml`).

> electron-updater reads from **published** (non-draft) releases. The release is created as a draft,
> so updates only roll out once you publish it. To test an upgrade: ship `v0.3.0`, install it, then
> publish `v0.3.1` and confirm the running `v0.3.0` shows the update banner.

## Code signing

**Current policy: ship unsigned.** No code-signing certificate is configured, so Windows builds are
unsigned.

- **Windows**: the unsigned NSIS/MSI installers run, but SmartScreen shows a "Windows protected your
  PC / unknown publisher" prompt on first install — the user clicks **More info → Run anyway**. In-app
  auto-update still works (it inherits the same SmartScreen prompt).
- **macOS / Linux**: not published by the current release profile. The local `electron:build:mac` and
  `electron:build:linux` scripts are unsupported convenience commands, not release outputs.

### Enabling signing later (optional)

electron-builder reads signing material from environment variables, so signing is opt-in: populate
the (currently commented) `env:` block on the "Build Electron app" step in `build-electron.yml` from
repository secrets.

- **Windows**: `CSC_LINK` (base64 `.pfx`) + `CSC_KEY_PASSWORD`. Removes the SmartScreen prompt.
  The admission gate's signing state machine requires real signature verification once credentials
  are configured — shipping with credentials but no verification fails the gate.
